A Vetting System for the Agentic Era
A proposal for how Proxify identifies the top 1% of engineers at scale — in a world where any candidate can generate syntactically perfect code in under twenty minutes.

The thesis: when AI can write the code, the test can no longer be the code. This proposal redesigns Proxify's evaluation pipeline from first principles — redefining what engineering excellence means, how it is measured, and how measurement is validated against real-world placement outcomes. Every stage, every score, and every circuit breaker in this system exists for one reason: to prove that what we select for in the funnel predicts what clients experience on day ninety.

01 · Problem framing
The Core Problem: Syntax is Dead

The existing foundational stages of elite screening map directly to the highest-validity predictors. However, when a candidate can generate syntactically perfect code via Copilot in under 20 minutes, the standard coding test stops measuring engineering excellence and starts measuring prompt fluency. The signal inverts.

1. The Signal Problem

Coding tests measure syntax execution. The AI era requires measuring judgment, system thinking, and the ability to validate AI output. Coding can be delegated; judgment cannot.

2. The Proof Problem

If the assessment doesn't mathematically predict client satisfaction downstream, the value proposition is unvalidated. We must install a continuous feedback loop into Eros.

3. The Scale Problem

At 150,000+ applications, human inconsistency compounds. Without calibration and automated circuit breakers, interviewer drift turns the funnel into a lottery.

One final note before the proposal begins. A system this detailed is only valuable if it can be implemented. The rollout sequence matters as much as the design. In the first 90 days: deploy Stage 1 written screens and Stage 2 sandbox for two tracks — Backend and Full-Stack — to establish baseline data while keeping live defense capacity manageable. In months three to six: add remaining tracks, begin collecting 90-day JPS from placed candidates, run first inter-rater calibration session. At six months: first quarterly refit of stage weights based on Pearson correlation data. The system reaches full operational confidence — all tracks, all stages, calibrated weights, validated circuit breakers — at approximately 12 months. Stakeholders who need to be aligned before launch: VP Assessment for rubric sign-off, Head of Talent Acquisition for stage sequencing integration with ATS, RnD leads per track for sandbox problem validation, and at least one client-side senior engineer per track for external content review.

02 · Framework Definition
The 6 Dimensions of Engineering Excellence

Before redesigning the funnel, we must define what we are actually measuring. Every stage in this system — every question, every sandbox bug, every live constraint — maps back to one or more of these six dimensions. A candidate's final score is not a single number. It is a vector across six axes. That vector is what Eros uses to match talent to clients. That vector is what 90-day job performance data validates or challenges quarter over quarter. These are not soft competencies. Each one is operationally defined, independently scored, and weighted by downstream predictive validity.

01
Architectural Judgment
The ability to decompose ambiguous systems, anticipate scaling bottlenecks, and reason about technical trade-offs before a single line of code is written.
Strong signal: names the failure mode before proposing the fix. Weak signal: jumps to implementation without articulating what they are solving or why.
02
Ambiguity Navigation
The ability to progress from a vague, unrefined business problem to a shipped solution without waiting for perfectly groomed requirements.
Strong signal: defines what they do not know before defining what they will build. Weak signal: treats every ambiguous problem as already defined and skips straight to solutions.
03
Agentic Fluency
The ability to safely orchestrate AI tools to accelerate output while critically auditing every suggestion for hallucinations, security flaws, and correctness.
Strong signal: modifies AI output, questions it, and times their acceptance deliberately. Weak signal: accepts suggestions in under three seconds without reading them — logged by the sandbox as a Blind Acceptance Event.
04
Async Collaboration
The ability to operate effectively across timezones through disciplined documentation, proactive communication, and maintaining high visibility without being asked.
Strong signal: structures written answers as if the reader has no context. Weak signal: assumes shared knowledge, skips reasoning, delivers conclusions without explanation.
05
Technical Articulation
The ability to translate complex architectural decisions and technical trade-offs into clear business language for non-technical stakeholders.
Strong signal: explains a database transaction constraint in terms of money and risk without using the word "transaction." Weak signal: responds to a business question with a technical answer the listener cannot act on.
06
Outcome Velocity
The ability to consistently prioritize business impact over technical completeness, shipping the right thing fast rather than the perfect thing late.
Strong signal: scopes a solution to the minimum that solves the stated problem and ships it. Weak signal: gold-plates the implementation, adds unrequested features, and misses the deadline.
03 · Funnel Design
The 5-Stage Vetting Pipeline

Most hiring funnels are elimination machines. Ours is a signal amplifier. Automation protects human bandwidth — so by the time a senior engineer sits across from a candidate, the conversation is never about syntax. It is about judgment. Stages 3, 4, and 5 below form the technical evaluation core. They are specified in full detail in the next section.

Funnel Trajectory Stage Conversion Pass Rate Programmatic Attrition Target Error Budget (Noise)
01 · Algorithmic Trajectory Screen 55.0% Move to Baseline 45.0% Profile Mismatch < 2.5% Resume Keyword Gaming
02 · Communication Baseline 36.3% Move to Written 63.7% Tone / Fluency Deficit < 1.0% Audio Environment Artifacts
03 · Written Diagnostic & Direction Screen 40.0% Move to Sandbox 60.0% Decomposition Failure < 4.0% Plausible-Sounding AI Prose

01
Algorithmic Trajectory Screen
Fully automated ATS parsing. Evaluates years of experience, tech stack depth, English proficiency signals, and market supply-demand ratio for the candidate's skill set. Documented top-decile contributors — active core maintainers of major open source packages, for example — receive an automatic fast-track flag that bypasses Stage 02 and routes them directly to Stage 04.
02
Communication Baseline
A 15-minute async video recording. Camera must be on. The candidate responds to one open-ended technical scenario and explains their reasoning out loud. Scored on linguistic clarity, structured articulation, and professional confidence by an AI audio and video parser. This stage does not auto-reject. It produces a Communication Score that is carried forward and weighted in the final composite. For client-facing roles, a low Communication Score triggers an escalation flag that the client engagement manager reviews before the Live Defense. Frequency of this stage is configurable per role — roles with high client communication requirements treat it as mandatory; roles that are primarily async and heads-down treat it as optional and data-collecting only.
03
Written Diagnostic and Direction Screen
Two async written questions with no interviewer present. The first tests technical decomposition — given a real production failure, walk through the exact diagnostic steps. The second tests direction-setting under ambiguity — given a vague business problem and no system access, define what you do not know, structure the investigation, and describe how you would orchestrate AI as part of it. Together these two questions measure the signal that implementation tests cannot: whether the candidate thinks before they build.
04
Guarded Sandbox
A 90-minute take-home project delivered inside a sandboxed IDE with full AI tool access. The candidate receives a broken codebase and must find and fix every bug. AI usage is not just allowed — it is tracked. The sandbox records every prompt sent, every suggestion received, and the time between receiving AI output and accepting it. A candidate who accepts AI suggestions in under three seconds without modification logs a Blind Acceptance Event. High counts collapse the Agentic Fluency score. What we measure is not whether they used AI. It is whether they understood what AI gave them.
05
Live System Defense
A 60-minute live video call with a senior Proxify engineer. No AI tools. No hints. The candidate opens with their submitted sandbox code and defends every decision. The interviewer then injects five compounding constraints one at a time — each one designed to break the candidate's existing design. Culture fit, remote-work dependability, and consulting posture are assessed here as scored dimensions of the live defense, not as a separate stage. Candidates who pass receive an evaluation vector that is automatically pushed to Eros, Proxify's AI matching engine, for immediate client ranking.

Frontend Expert: The Full Lifecycle Pipeline

Measurement Target: React reconciliation, rendering pipelines, ambiguity navigation, and main-thread execution performance. Technology Selection: Default to React; Next.js accepted as a bonus signal for SSR/hydration evaluation.

Stage 1: Async Screen 30 Mins • Format: Written
Diagnostic Interrogation

Question given to the candidate: A client's React checkout page freezes for 800ms every time a user clicks the "Pay Now" button. The conversion rate has dropped 12%. You have Chrome DevTools open. Walk me through exactly what you do, step by step, from the moment you reproduce the freeze.

Six dimensions scored 0–10 by a human reviewer using a fixed rubric:

  • D1 — Root cause identification (Wt 0.25): A strong candidate says: the main thread is being blocked during the click handler — likely a synchronous heavy computation, a layout thrash from reading and writing DOM properties, or a large re-render cascade. A weak candidate says "it is probably a slow API call" without touching the rendering pipeline.
  • D2 — Correct tooling named (Wt 0.15): We expect: Chrome Performance tab to capture a trace, React Profiler to identify which components re-rendered, and the INP metric.
  • D3 — Rendering model understood (Wt 0.25): Expect explanation of React reconciliation. If they cannot explain why a `setState` call causes a synchronous render before paint, they fail.
  • D4 — Concrete fix proposed (Wt 0.20): Acceptable: move heavy computation into `useMemo`, break calculation into chunks using `requestIdleCallback`, or offload to a Web Worker.
  • D5 — Secondary risks raised (Wt 0.10): Mentioning context splitting or `useReducer` as structural fix is strong.
  • D6 — Communication structure (Wt 0.05): Does it go: reproduce, trace, identify, fix, verify?
Score formula: S1 = (D1×0.25 + D2×0.15 + D3×0.25 + D4×0.20 + D5×0.10 + D6×0.05) × 10. Max S1 = 10.
Circuit Breaker CB1: IF D3 < 7 OR D4 < 7 → DISQUALIFY. Do not proceed to Stage 1B.
Pass threshold: Total 42 out of 60 or higher. D3 and D4 must each be 7 or above.

05 · AI in the System
Three Angles on AI

The assignment asked for three distinct positions on AI. They are addressed here directly, before the technical specifications, because every design decision in Section 04 follows from the positions taken here.

A
Candidate AI Usage
How candidates use AI is not the threat. Blind trust in AI output is the threat. The system does not restrict AI — it measures the relationship between the candidate and the AI. Every sandbox session tracks Time to Accept and Modification Rate. A candidate who pastes AI output without reading it fails not because they used AI but because they demonstrated they cannot validate what AI gives them. That skill — critical review of AI output — is the exact skill the highest-performing engineers of the next decade will need. Restricting AI in assessment is like testing driving skills in a parking lot with the engine off.
B
AI as Part of the Funnel
Three places where AI improves the funnel itself. First: the ATS screen uses AI to parse trajectory signals that a keyword filter cannot — identifying engineers whose GitHub contributions show systems thinking even when their CV uses different terminology. Second: Stage 1 written responses are AI-scored against a fixed rubric as a triage layer before human review — this reduces reviewer load by approximately 60% while maintaining calibration because the rubric dimensions are objective enough for reliable automated scoring. Third: the communication baseline video is AI-parsed for linguistic clarity and structured articulation, producing a score that feeds into the final composite without requiring a human to watch every recording. What AI does not do: make final hiring decisions, score the live defense, or interpret ambiguous answers at the boundary of pass and fail thresholds. Those decisions require a human.
C
Failure Modes and Gaming
Four ways this system can be broken and what the design does about each. First: a candidate who has seen these exact broken codebases before will score well on pattern recognition, not engineering judgment — mitigation is quarterly rotation of the sandbox problem. Second: the TTA metric can be gamed by waiting artificially before accepting AI output — the system cross-references TTA against MR, so waiting thirty seconds then accepting with zero modifications scores the same as accepting in two seconds. Third: the written direction screen can be answered with AI-generated frameworks that sound structured but contain no genuine reasoning — human reviewers are calibrated specifically to recognize generic frameworks with no specific hypothesis, no named diagnostic tool, no concrete question. Fourth: the live defense cannot be gamed because the constraints are injected in real time and are designed to break whatever the candidate submitted.

The system is not designed to catch cheaters. It is designed so that cheating does not produce the signal we are looking for.

08 · Measurement & Predictive Validity
How We Know the Funnel Is Working

This section answers the question the JD asked most directly: how do you use data to validate that the assessment is reliable, accurate, and genuinely predictive of on-the-job performance? Three layers of measurement are required. The first validates that each stage measures what it claims to measure. The second validates that the stages predict downstream performance. The third validates that the system improves over time.

01
Inter-Rater Reliability
Stage 1 and Stage 1B scores are assigned by human reviewers. If two reviewers score the same response differently, the dimension is not reliably measured — and an unreliable measure cannot be predictive. We track Cohen's Kappa for every scored dimension across all reviewers every quarter. Target κ ≥ 0.70 for each dimension before it is used in a pass/fail decision. Dimensions scoring below κ = 0.60 are pulled from automated decision-making and flagged for rubric revision. Calibration sessions use a set of twenty anchor responses — five per score tier — that all reviewers must score before rating live candidates in a new quarter.
Signal: If κ drops below 0.60 on D1 or D3 in any track, the rubric is ambiguous and the circuit breaker for that dimension must be suspended until the rubric is revised and recalibrated.
02
Predictive Validity Against 90-Day Performance
The funnel score F is meaningless unless it predicts something real. We collect a Job Performance Score from every placed candidate's manager at 90 days. JPS is a weighted composite of manager rating, code review pass rate, production incident rate, and a track-specific signal such as cross-layer bug rate for full-stack engineers or pipeline reliability rate for data engineers. We compute r = Pearson(F, JPS) monthly once n ≥ 30. Target r ≥ 0.55 at six months. If r < 0.40 at six months, the stage weights are wrong — we run a quarterly refit using each stage's individual correlation with JPS as its new weight.
Signal: r ≥ 0.55 is the minimum acceptable threshold. r ≥ 0.70 means the system is strong and stage weights are approximately correct. r < 0.40 means the system is measuring something other than what predicts job performance and a structural review is required.
03
False Positive and False Negative Tracking
A false positive is a candidate who scored F ≥ 82 and received JPS < 5 at 90 days — the funnel passed someone who failed. A false negative proxy is a candidate who scored F < 65 and appeared in a strong senior engineering role at another company within six months — the funnel rejected someone who succeeded elsewhere. We track FPR and FNR monthly. Target FPR < 0.10 and FNR < 0.15. If FPR exceeds 0.15 for two consecutive quarters, Stage 2 circuit breakers must be tightened. If FNR exceeds 0.20, Stage 1 rubric thresholds are too aggressive and must be reviewed.
Signal: FPR is the more damaging error for Proxify — a client who receives a weak engineer loses trust in the network. FNR is damaging for supply — we are filtering engineers we should be placing. Both must be tracked independently and weighted differently in the operational response.
04
A/B Testing the Funnel Itself
The funnel is not static. We run controlled experiments on individual stages to test whether changes improve predictive validity. Example: if we propose adding a fourth live defense constraint, we run it on 20% of candidates for one quarter and compare FPR, FNR, and r against the control group before rolling out. Experiments are never run during ramp-up periods — the first six months of operation use the default design to establish a baseline. Changes are validated with n ≥ 50 per arm before any weight or threshold adjustment is applied to the full funnel.
Signal: No change to the funnel — no new stage, no rubric revision, no weight adjustment — is made without a validated experiment or a quarterly refit based on n ≥ 50 downstream performance observations.
05
External Expert Calibration
Internal reviewers develop blind spots. Every six months, a panel of three to five external senior engineers — sourced from Proxify client companies and academic institutions with assessment expertise — reviews a sample of fifty scored responses across all tracks. They score the responses independently against the same rubric and their scores are compared against the internal baseline. Divergences above 15% on any dimension trigger a rubric review. For the data and AI track specifically, we engage an external ML practitioner to validate that the broken code problems reflect real production failure patterns, not academic exercises. For the predictive validity model, we engage a statistician to review the JPS formula and correlation methodology annually.
Signal: External review is not optional — it is the check on the check. Internal calibration maintains consistency. External calibration maintains accuracy.
06
Systemic Process Redesign Triggers
When a track's mathematical weight variance fails to stabilize, or when downstream client satisfaction reports flag performance churn, numbers alone are insufficient. If a track's Pearson correlation drops below r = 0.35, or if client-side placement rejections spike past 15% within a single quarter, the system declares a structural assessment breach. This trigger bypasses standard weight recalculation and mandates a clean-slate overhaul of the sandbox environment, an absolute rotation of the core codebase vulnerabilities, and a complete re-engineering of the live interview constraints to align back with evolving production conditions.
Signal: Client rejections are the ultimate verification constraint. If the funnel scores match on paper but fail in client teams, the process is structurally compromised and a clean-slate rebuild fires automatically.
09 · Scaling Architecture
Scalability, Calibration & Operations

Processing 150,000+ applicants per year requires human effort to scale sub-linearly. By automating Stages 1 through 3, we protect valuable human resources.

1. Automated Screening (Stages 1-3)

Filters the initial 150,000+ candidates down to roughly 12,000 without using a single hour of human engineering time. Evaluates candidates programmatically against the telemetry matrices.

2. Human-in-the-Loop (Stages 4-5)

Human engineering bandwidth is deployed exclusively at the final stages. Because the automated circuit breakers filter out 92% of the noise, senior engineers only spend time interviewing candidates who survived the deep-audit sandbox.

Interviewer Calibration Engine

With a pool of 100+ senior evaluators, scaling human steps risks rating drift. We mandate that 5% of Stage 5 Live Defenses are dual-graded (shadowed). If any interviewer's ratings deviate by more than +/-12% from the track cohort mean, the system triggers an automatic alignment flag and routes both graders to a calibration workshop.

10 · Honest Assessment
Where This System Can Fail

The assignment specifically asked for trade-offs and downsides. They are listed here without softening, because a system that cannot name its own weaknesses is not a system anyone should trust.

01
The Pattern Recognition Problem
The sandbox problems are specific enough that a candidate who has seen similar broken codebases in a bootcamp or previous assessment will score well without understanding the underlying principles. The surface behaviors — correct fixes, clean diffs, passing tests — are identical to genuine understanding. Mitigation exists in the quarterly rotation and the live defense, but it is imperfect. A candidate who pattern-matched Stage 2 will be exposed in Stage 3. The system relies on this exposure being reliable. If candidates can also pattern-match the live defense constraints, the circuit breakers in Stage 3 need to be made stricter, not the sandbox harder.
02
The Human Reviewer Calibration Problem
Stage 1 and Stage 1B scores are assigned by human reviewers using fixed rubrics. At 150,000 candidates per year, even small calibration drift between reviewers compounds into a meaningful false negative rate. The mitigation is quarterly calibration sessions using scored anchor responses and human confirmation on borderline rejections. But this is operationally expensive. If reviewer bandwidth becomes a bottleneck, the temptation will be to automate Stage 1 scoring entirely. That would be a mistake — the dimensions that matter most, specifically problem decomposition and outcome framing, are exactly the dimensions that resist reliable automated scoring because they require judgment about what the candidate chose not to say.
03
The Communication Score Trade-off
The video communication baseline is configurable per role. Making it mandatory for all roles would filter out strong engineers who are slightly less polished communicators but who could grow into client-facing positions. Making it optional risks inconsistency — some clients receive candidates whose communication was never assessed. The current design accepts this trade-off in favor of not over-filtering a supply-constrained talent pool. It should be revisited once the system has enough data to correlate communication scores with 90-day client satisfaction ratings directly.
04
The 90-Day Feedback Loop Latency
The entire predictive validity model depends on collecting job performance scores at 90 days and feeding them back into stage weight calibration. This means the system will run with uncalibrated default weights for the first three to six months of operation, during which it will produce some false positives and false negatives that a calibrated system would not. The target is r ≥ 0.55 Pearson correlation between funnel score and 90-day JPS once n ≥ 30. Until that threshold is reached, human review is required on all borderline rejections. The system is designed to improve over time. It is not optimized at launch.
